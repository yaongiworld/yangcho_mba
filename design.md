## Problem Statement

Yangcho is a cosmetics R&D researcher at a large Korean beauty company preparing for MBA applications. She needs a personal project that demonstrates both technical execution (AI/data science) and business thinking (market analysis, product strategy). The project: an AI-powered tool that mines US consumer complaint data to identify formulation gaps that K-Beauty R&D can address, then auto-generates product concept briefs. A weekly discovery pipeline keeps the tool alive and generating new insights over time.

## What Makes This Cool

The killer feature is the **formulation science translation layer**. Every beauty market intelligence tool stops at sentiment ("people don't like this sunscreen"). This tool goes three levels deeper: sentiment → formulation root cause ("film former compatibility issue") → K-Beauty ingredient solution → complete product concept brief. Only someone with actual R&D chemistry expertise could build this bridge. The tool is proof that AI is necessary but insufficient without domain expertise.

The "living portfolio" angle — a weekly pipeline that keeps surfacing new white spaces — makes the MBA demo unforgettable. "It's still running and it found two new product opportunities this week" is a sentence no other applicant can say.

## Constraints

- **Timeline:** Must be complete before MBA application deadlines. Core build: 5-6 weekends. Buffer: 2 additional weekends for polish and unexpected issues. Total: 7-8 weekends.
- **Team:** Yangcho (R&D expertise, product vision) + husband (CTO, engineering execution).
- **Budget:** Minimal — Claude/GPT API costs (~$30-50/month), Streamlit Cloud free tier.
- **Secrecy:** Being prepared without employer knowledge. No proprietary company data can be used.
- **Data:** Public US consumer data only (Amazon reviews via Hugging Face, Reddit via PRAW, public social media).
- **Tech stack:** Python, Streamlit, Claude/GPT API, SQLite.

## Premises

1. US consumer complaint data (reviews, Reddit, social media) contains enough signal to identify real formulation gaps — not just marketing complaints — that K-Beauty R&D can address.
2. Formulation expertise is the key differentiator: the tool is impressive BECAUSE it translates consumer language into R&D-actionable insights, which a generic AI cannot.
3. For MBA admissions, a polished dashboard with 2-3 hero case studies is more compelling than a feature-complete platform. Quality of insight > breadth of data.
4. Primary audience is admissions committees wanting business thinking + technical execution. Does NOT need enterprise scale.
5. Distribution doesn't matter — portfolio demo, not a shipped product. Hosted Streamlit app or recorded demo video is sufficient.

## Approaches Considered

### Approach A: "Polish & Case Study" (Minimal)
Keep existing prototype. Add 3 deeply researched case studies with formulation annotations. Polish UI. Effort: S (1-2 weekends). Risk: Low. Undersells the expertise and doesn't demonstrate the full pipeline vision.

### Approach B: "Knowledge Graph + Auto-Brief" (Ideal)
Build formulation knowledge base (50-80 entries). Add RAG pipeline for auto-generating K-Beauty Product Briefs. Confidence score. 3 hero case studies + live cluster exploration. Effort: M (3-4 weekends). Risk: Medium.

### Approach C: "The Living Portfolio" (Original choice — superseded 2026-05-02)
Everything in Approach B, plus a weekly discovery pipeline that scrapes new complaints, surfaces new white spaces automatically, and includes a timeline view showing discoveries over time. Effort: L (5-6 weekends). Risk: High (scraping reliability).

### Approach D: "The Hybrid" (Chosen 2026-05-02 via CEO review)
Everything in Approach C — knowledge base, auto-brief generator, dashboard, weekly cron, discovery timeline — but **publishing is editorial, not automatic**. New auto-generated briefs land in a `pending_review` queue. Yangcho approves ~1 discovery per month before it surfaces on the public Discovery Timeline. The cron can fail silently without breaking the live demo. Effort: 6 core engineering weekends + 1 white paper weekend + 1 buffer = 8 total (same headline as C, with 1 buffer absorbed by the white paper deliverable; engineering work itself is smaller than C). Risk: Medium. Full breakdown in `~/.gstack/projects/yangcho_mba/ceo-plans/2026-05-03-white-space-miner.md`.

Why this won the CEO review:
- The "still running, still discovering" narrative is the differentiator — and it's preserved.
- A monthly cadence (1 curated discovery/month) reads as more credible than weekly auto-spam.
- During application season, operational *attention* is the scarcest resource, not engineering time.
- A silently broken cron no longer breaks the demo. Yangcho stays in editorial control.

### Implementation Plan (6 Core Weekends + 2 Buffer)

**Weekend 1: Formulation Knowledge Base (THE MOST IMPORTANT WEEKEND)**
- Write 50-80 entries mapping complaint keywords → formulation root causes → K-Beauty solutions
- Format: JSON/YAML with fields: complaint_terms[], root_cause, category, kbeauty_approach, hero_ingredients[], difficulty, kbeauty_solvability (1-5), competitive_emptiness (1-5), references
- **Version control:** The knowledge base JSON file lives in the Git repo alongside the code. All changes are tracked via commits, providing full revision history and rollback capability.
- This is Yangcho's job, not the CTO's. It requires R&D chemistry expertise.
- Test: can the LLM use this knowledge base to correctly translate 10 sample complaints?
- **Ongoing quality gate:** For the weekly pipeline, Yangcho reviews all new auto-generated briefs (~10-20 min/week) before they appear on the dashboard. No brief goes live without human sign-off.

**Weekend 2: Data Pipeline + Clustering**
- Pull 10K+ skincare/sunscreen reviews from Hugging Face Amazon Reviews dataset
- Pull top 1000 posts from r/SkincareAddiction, r/AsianBeauty, r/MakeupAddiction via PRAW
- **Filtering step:** Not all posts are complaints. Filter for negative sentiment / problem language using keyword heuristics + LLM classification. Expected yield: ~30-40% of Reddit posts and ~20-30% of reviews contain actionable complaint content. Estimate ~4-5K usable complaint data points.
- Generate embeddings (OpenAI text-embedding-3-small), cluster with HDBSCAN
- **Cluster labeling:** For each cluster, sample the 10 most central complaints (nearest to centroid). Prompt Claude to read these samples, match against the formulation knowledge base, and assign the most likely formulation root cause category. If no match, flag as "unclassified" for Yangcho to review and potentially expand the knowledge base.
- Store in SQLite

**Weekend 3: Auto-Brief Generator**
- Build RAG pipeline: for each complaint cluster, query formulation knowledge base, generate a structured Product Brief via Claude/GPT API
- Implement Confidence Score with defined inputs:
  - **complaint_volume** (0.4 weight): Normalized count of complaints in this cluster. Scale 1-5, auto-calculated from data.
  - **kbeauty_solvability** (0.35 weight): How well K-Beauty techniques address this gap. Scale 1-5, manually rated per entry in the knowledge base by Yangcho. 5 = K-Beauty has a proven superior approach, 1 = no clear K-Beauty advantage.
  - **competitive_emptiness** (0.25 weight): How few US market products currently address this gap. Scale 1-5, manually rated per entry in the knowledge base. 5 = no products on shelf, 1 = saturated. Yangcho rates these based on her market knowledge; not automated.
  - Formula: score = (volume * 0.4) + (solvability * 0.35) + (emptiness * 0.25)
  - **Score tiers:** Above 3.5 = high-confidence opportunity (highlighted in dashboard). 2.5-3.5 = moderate. Below 2.5 = low-confidence (shown but deprioritized).
- Generate briefs for all clusters, rank by confidence score
- Select top 3 as hero case studies
- **Validation step:** Yangcho spot-checks the top 10 auto-generated briefs for formulation accuracy before publishing to dashboard. This is the human-in-the-loop quality gate.

**Weekend 4: Dashboard Build**
- Page 1: Interactive Plotly scatter — clusters color-coded by formulation category, sized by complaint volume. Click any cluster.
- Page 2: Product Brief view — formatted brief with problem, root cause, proposed solution, confidence score
- Page 3: Hero Case Studies — 3 polished cards with deep narrative
- Page 4: Discovery Timeline (placeholder for now)

**Weekend 5: Weekly Pipeline + Timeline**
- **Execution environment:** The weekly pipeline runs as a cron job on a personal machine (or GitHub Actions on a weekly schedule), NOT on Streamlit Cloud. Streamlit Cloud hosts the dashboard only. The pipeline writes results to SQLite, which is committed to the repo or synced to the dashboard.
- Build scraping pipeline: weekly cron that pulls new Reddit posts + checks for new review data
- **Fallback if Reddit API access is lost:** Switch to periodic manual export via browser (Reddit search → copy/paste), or pivot to monitoring public Sephora/Ulta review pages via simple HTTP requests. The pipeline should be designed to accept CSV/JSON input from any source, not tightly coupled to PRAW.
- **New pattern detection (v1 approach):** Generate embeddings for new complaints. Compute cosine similarity to existing cluster centroids. Complaints below a 0.7 similarity threshold to ALL existing centroids are flagged as potential new patterns. Flagged items are written to a `pending_review` table in SQLite and displayed on a hidden `/review` page in the Streamlit app (password-protected or URL-only, not linked from the public dashboard). Yangcho reviews this page weekly (~10-20 min) and decides: new cluster, merge into existing, or noise. This is deliberately simple — sophistication isn't needed at portfolio scale.
- Generate briefs for confirmed new discoveries
- Timeline view: chronological feed showing "Week of April 20: discovered X new white space"
- Deploy dashboard to Streamlit Cloud

**Weekend 6: Polish + Demo Prep**
- UI polish: consistent styling, professional typography (desktop-only — admissions committees review on laptops, mobile is unnecessary)
- Record a 2-minute demo video walkthrough
- Stress-test the weekly pipeline (run it 2-3 times manually, verify new pattern detection works)
- Prepare MBA essay language referencing the project

**Nice-to-have (buffer weekends 7-8 if time allows):**
- Write a companion blog post / LinkedIn article explaining the methodology
- Add a "Methodology" page to the dashboard explaining the pipeline for curious reviewers

### Key Technical Decisions

- **SQLite over Postgres**: Good enough for portfolio scale, zero ops burden
- **Claude API over fine-tuned model**: Better reasoning for complex formulation translation, no training data needed
- **HDBSCAN over K-means**: Handles variable-density clusters, doesn't require pre-specifying k
- **JSON knowledge base over vector DB**: Starts simple, the 50-80 entries fit easily in context window for RAG
- **Streamlit Cloud for hosting**: Free, zero-config, shareable URL for MBA applications
